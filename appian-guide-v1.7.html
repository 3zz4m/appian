SECTION 01Core Concepts

Before touching a single command, make sure you have these concepts solid. Every production problem traces back to one of them.

1.1What is Appian & Why OpenShift?

Appian is an enterprise low-code platform for process automation and AI-powered workflows. Since version 25.4, Appian runs exclusively as a containerised workload on Kubernetes. OpenShift is the enterprise-grade, security-hardened Kubernetes distribution used in production.

All Appian microservices run as containerized pods managed by the Appian Operator — a Kubernetes-native controller with automated lifecycle management.

SECTION 02First Login & Cluster Access

Before running a single oc command you need an authenticated session. This section gets you from zero to a working terminal session with cluster-admin access.

2.1Get Your API URL & Credentials

SourceWhere to Find
IPI installLast lines of openshift-install create cluster output. Also saved to auth/kubeconfig and auth/kubeadmin-password
UPI / existing clusterAsk your OCP platform team for the API URL (https://api.<cluster>.<domain>:6443) and a service account token or username/password
OCP ConsoleTop-right → your username → "Copy login command" — pastes a ready-to-run oc login command with token

2.2Log In with oc login

bash
# Option A: username + password
oc login https://api.<cluster>.<domain>:6443 \
  --username=kubeadmin \
  --password="<kubeadmin-password>"

# Option B: token (from console "Copy login command")
oc login --token=sha256~xxxx \
  --server=https://api.<cluster>.<domain>:6443

# Air-gap: API cert may be self-signed — add CA or skip verify (dev only)
oc login https://api.<cluster>.<domain>:6443 \
  --certificate-authority=/path/to/cluster-ca.crt \
  --username=kubeadmin --password="<password>"
AIR-GAPNever use --insecure-skip-tls-verify in production. Import the cluster CA instead: oc config set-cluster <name> --certificate-authority=/path/to/ca.crt.

bash · verify-access.sh

SECTION 04Prerequisites

Missing a prerequisite will silently block you hours into deployment. Verify every item before running a single command.

2.1Infrastructure Checklist

  • ☐Red Hat OpenShift 4.14+ cluster operational (bare-metal IPI/UPI or CRC for dev)
  • ☐Minimum 3 worker nodes (HA) or 1 node (SA/CRC) with 16+ vCPU and 64+ GB RAM each
  • ☐MetalLB or equivalent load balancer configured (bare-metal)
  • ☐cert-manager operator installed and running
  • ☐ODF (OpenShift Data Foundation) installed for CephFS/RBD storage (HA only)
  • ☐DNS configured for Appian URL and OpenShift routes
  • ☐TLS certificates provisioned (enterprise CA or self-signed for dev)

2.2Accounts & Downloads

  • ☐Red Hat account at console.redhat.com
  • ☐MyAppian account with active subscription
  • ☐AoK licence files requested (temporary 6-day + long-term)
  • ☐Appian Operator Helm chart downloaded (.tgz)
  • ☐Appian container images available (operator, webapp, data-server, search-server, service-manager, kafka, zookeeper, httpd)
  • ☐appian.yaml CR template downloaded from Appian docs

2.3External Database

  • ☐SQL Server 2022 (or MariaDB/MySQL/PostgreSQL) installed and running
  • ☐Appian database created with collation SQL_Latin1_General_CP1_CI_AS
  • ☐READ_COMMITTED_SNAPSHOT ON set on database
  • ☐DB user with db_owner for initial schema creation
  • ☐Database is network-reachable from pod CIDR on port 1433
  • ☐JDBC connection URL confirmed working
CRITICALLicence files are site-specific. k3.lic and k4.lic must match the site name and URL in your CR. A mismatch prevents the site from starting.
UPGRADE PREREQUISITEBefore any Appian Operator upgrade, pause the MachineConfigPool to prevent OCP from applying node reboots simultaneously. Double disruption (operator restart + node drain) is the most common cause of extended HA downtime during maintenance windows.

Resume after: oc annotate mhc appian-worker-mhc -n openshift-machine-api cluster.x-k8s.io/paused-

2.4Tools

  • ☐oc CLI installed and logged in to cluster
  • ☐Helm 3 installed
  • ☐Podman or Docker available for image management
  • ☐skopeo installed (air-gapped environments)

2.6Preflight Validation Script

Run this script before starting any deployment. All checks must pass:

bash · appian-preflight.sh
#!/bin/bash
# Appian on OpenShift — Pre-Deployment Health Check
echo "=== APPIAN OPENSHIFT PREFLIGHT ==="
# 1. Cluster connectivity
oc whoami >/dev/null 2>&1 || { echo "❌ Not logged in"; exit 1; }
echo "✅ Cluster: $(oc whoami --show-server)"
# 2. Storage classes
oc get storageclass --no-headers | grep -q . || { echo "❌ No StorageClass"; exit 1; }
echo "✅ StorageClasses: $(oc get sc --no-headers | wc -l) available"
# 3. cert-manager
oc get pods -n cert-manager --no-headers 2>/dev/null | grep -q Running || { echo "❌ cert-manager not running"; exit 1; }
echo "✅ cert-manager: Running"
# 4. Database reachable
SQL_HOST=${SQL_HOST:-"sql-server.example.com"}
SQL_PORT=${SQL_PORT:-1433}
nc -zv $SQL_HOST $SQL_PORT 2>&1 | grep -q succeeded || { echo "❌ DB unreachable"; exit 1; }
echo "✅ Database: $SQL_HOST:$SQL_PORT reachable"
# 5. DNS resolution
APPIAN_URL=${APPIAN_URL:-"appian.yourcompany.com"}
dig +short $APPIAN_URL | grep -q . || { echo "❌ DNS failure for $APPIAN_URL"; exit 1; }
echo "✅ DNS: $APPIAN_URL resolves"
# 6. Helm available
helm version --short >/dev/null 2>&1 || { echo "❌ Helm not installed"; exit 1; }
echo "✅ Helm: $(helm version --short)"
echo "✅ ALL PREFLIGHT CHECKS PASSED"

SECTION 06MachineConfig & Node Tuning

OCP defaults are tuned for general workloads. Appian requires specific kernel and OS settings applied via MachineConfig — the only supported way to tune nodes in OCP.

CRITICALEach MachineConfig application triggers a rolling node reboot via MachineConfigDaemon. Apply during a maintenance window and watch with oc get machineconfigpool -w.

28.1Kernel Tuning — vm.max_map_count & File Limits

yaml · appian-worker-tuning.yaml
apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfig
metadata:
  name: 99-appian-worker-tuning
  labels:
    machineconfiguration.openshift.io/role: worker
spec:
  config:
    ignition:
      version: 3.4.0  # OCP 4.14+ — use 3.4.0 (3.2.0 triggers deprecation warning)
    storage:
      files:
      - path: /etc/sysctl.d/99-appian.conf
        contents:
          source: data:,vm.max_map_count%3D262144%0Afs.file-max%3D131072%0Anet.core.somaxconn%3D32768%0Avm.swappiness%3D1
        mode: 0644
WHYvm.max_map_count=262144 is required by Appian Search Server (Elasticsearch). Default is 65530 — search-server will log a critical warning and may refuse to start.

28.2Transparent Hugepages — Disable for Kafka

yaml · disable-thp.yaml
apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfig
metadata:
  name: 99-appian-disable-thp
  labels:
    machineconfiguration.openshift.io/role: worker
spec:
  config:
    ignition:
      version: 3.4.0  # OCP 4.14+ — use 3.4.0 (3.2.0 triggers deprecation warning)
    systemd:
      units:
      - name: disable-thp.service
        enabled: true
        contents: |
          [Unit]
          Description=Disable Transparent Huge Pages
          [Service]
          Type=oneshot
          ExecStart=/bin/sh -c 'echo never > /sys/kernel/mm/transparent_hugepage/enabled'
          ExecStart=/bin/sh -c 'echo never > /sys/kernel/mm/transparent_hugepage/defrag'
          [Install]
          WantedBy=multi-user.target

28.3Verify Tuning Applied on Node

bash
oc debug node/<worker> -- chroot /host sysctl vm.max_map_count fs.file-max
# Expected: vm.max_map_count = 262144
oc debug node/<worker> -- chroot /host cat /sys/kernel/mm/transparent_hugepage/enabled
# Expected: always madvise [never]
oc get machineconfigpool worker -o jsonpath='{.status.conditions[?(@.type=="Updated")].status}'
# Expected: True
SECTION 07RBAC Least Privilege

The guide uses cluster-admin throughout for clarity. Production clusters need a tiered RBAC model so developers and L1 support cannot accidentally destroy production resources.

SECURITYGiving cluster-admin to every team member who follows this guide is a critical security gap in enterprise environments. Implement these roles before granting access to production.

yaml · appian-platform-admin-role.yaml
apiVersion: rbac.authorization.k8s.io/v1

  name: appian-platform-admin
rules:
- apiGroups: [crd.k8s.appian.com]

- apiGroups: [""]

  verbs: [get, list, watch, create, update, patch, delete]

  verbs: [get, list, watch, update, patch]

SECTION 08OLM Catalog Mirroring

In an air-gapped cluster, OperatorHub shows zero operators. You must mirror the Red Hat operator catalog before installing cert-manager, MetalLB, Grafana Operator, or OADP/Velero. This is a prerequisite for everything in Section 12 onwards.

BLOCKERcert-manager is required before the Appian Operator can be installed (Section 12). cert-manager is installed via OLM. Without catalog mirroring, cert-manager installation will fail with "no package cert-manager-operator in any channel".

6.1ImageSetConfiguration for Required Operators

yaml · imageset-config.yaml
apiVersion: mirror.openshift.io/v1alpha2
kind: ImageSetConfiguration
storageConfig:
  local:
    path: /tmp/oc-mirror-metadata
mirror:
  operators:
  - catalog: registry.redhat.io/redhat/redhat-operator-index:v4.16
    packages:
    - name: cert-manager-operator

SECTION 32AI Configuration

Appian 26.x ships with AI Copilot, Intelligent Document Processing (IDP), and AI Skills. All require explicit configuration — they are disabled by default.

bash

18.2AI Custom Properties (CR) — Appian 26.x Correct Keys

CRITICALIn Appian 26.x, AI is configured via Admin Console → AI Features, not CR custom properties. The CR is used only to inject the API key secret reference. Using wrong property keys silently fails.
yaml · correct CR env injection for 26.x
spec:
  webapp:
    env:
      # Inject API key from secret — correct pattern for 26.x
    - name: APPIAN_OPENAI_API_KEY
      valueFrom:
        secretKeyRef:
          name: appian-ai-secret
          key: openai-api-key
    customProperties:
      # Azure OpenAI endpoint (if using Azure)
      conf.suite.ai.AZURE_OPENAI_ENDPOINT: https://<your-resource>.openai.azure.com
      conf.suite.ai.AZURE_OPENAI_API_VERSION: 2024-02-01
      conf.suite.ai.AZURE_OPENAI_DEPLOYMENT_ID: gpt-4o
TIPAfter CR apply, go to Admin Console → AI Features to enable Copilot, IDP, and AI Skills individually. Each feature must be toggled on separately.

18.3Air-Gapped AI — Outbound Proxy

SECTION 33Mobile App Config

The Appian mobile app (iOS/Android) connects to your OCP-hosted Appian via the Mobile Gateway. Misconfiguration is the #1 cause of "app cannot connect" issues.

22.1Mobile Gateway URL Configuration

CRITICALThe mobile gateway URL must match conf.suite.SERVER_AND_PORT exactly, including the port. A mismatch causes silent authentication failures in the mobile app.
yaml · mobile customProperties
spec:

      conf.suite.SCHEME: https

      # Mobile-specific: deep link scheme for iOS/Android
      conf.mobile.deepLinkScheme: appian
      # Enable push notifications (requires APNs/FCM config)
      conf.mobile.pushNotifications.enabled: true
